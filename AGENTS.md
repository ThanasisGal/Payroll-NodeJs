# AGENTS.md — Οδηγίες για το Payroll-NodeJs

## Γενικό πλαίσιο project

Το project είναι εφαρμογή μισθοδοσίας και υποβολών ΕΡΓΑΝΗ.

Τεχνολογίες:

* Node.js / Express
* EJS views
* MongoDB / Mongoose
* Bootstrap 5
* PM2 και nginx σε production
* Τοπική ανάπτυξη σε Ubuntu

## Απόλυτοι κανόνες ασφαλείας

1. Μην κάνεις deploy σε production.
2. Μην τρέχεις καταστροφικές εντολές.
3. Μην πειράζεις `.env` αρχεία.
4. Μην αλλάζεις credentials, passwords ή API keys.
5. Μην κάνεις αλλαγές σε production/minified αρχεία, όπως `public/min.js`, εκτός αν ζητηθεί ρητά.
6. Κάνε μικρές, στοχευμένες αλλαγές.
7. Πριν αλλάξεις κώδικα, διάβασε τα σχετικά controller, model, route και frontend αρχεία.
8. Μετά από κάθε αλλαγή, εξήγησε ακριβώς ποια αρχεία άλλαξαν και γιατί.
9. Πάντα να δείχνεις `git diff --stat`.
10. Μην αλλάζεις dependencies/package.json αν δεν είναι απολύτως απαραίτητο και χωρίς εξήγηση.
11. Μην ξαναγράφεις μεγάλες άσχετες περιοχές κώδικα.
12. Μην αλλάζεις υπολογισμούς μισθοδοσίας χωρίς να εξηγήσεις ακριβώς τη φόρμουλα που άλλαξες.

## Προτιμήσεις χρήστη

* Ο χρήστης θέλει πλήρη αρχεία/πλήρεις αλλαγές, όχι placeholders.
* Για local development να μη συμπεριλαμβάνονται minified/public build files.
* Να δίνονται πρακτικές οδηγίες και όχι γενικές θεωρίες.
* Να αποφεύγονται αλλαγές που δεν ζητήθηκαν.

## Ειδικοί κανόνες Payroll

* Για πεδία απολογιστικών υπολογισμών να χρησιμοποιούνται τα `*_apologistika`.
* Η ονομασία `nominhs` να παραμείνει προσωρινά όπου ήδη χρησιμοποιείται για νόμιμες υπερωρίες, εκτός αν ζητηθεί ρητή μετονομασία.
* Το `ores_argion` να αντιστοιχίζεται στο `ores_argion_prosayxhsh_apologistika` όπου χρειάζεται.
* Για `hmeres_mh_ergasias`:

  * `kathgoria_ergasias_apologistika = "ΕΡΓ"` σημαίνει εργάσιμη ημέρα
  * `kathgoria_ergasias_apologistika = "ΜΕ"` σημαίνει μη εργάσιμη ημέρα
  * `kathgoria_ergasias_apologistika = "ΑΝ"` σημαίνει ρεπό
  * `astheneia_apologistika = true` σημαίνει ασθένεια
* Απόφυγε αλλαγές που δημιουργούν πολλαπλά loader flickering.
* Όπου γίνεται, να υπάρχει ενιαίο pipeline φόρτωσης/υπολογισμού και όχι πολλά ανεξάρτητα change events.
* Μην αλλάζεις payroll formulas χωρίς ρητή εντολή.

## ΕΡΓΑΝΗ / REST API

* Μην κάνεις πραγματική υποβολή στο ΕΡΓΑΝΗ.
* Μην καλείς εξωτερικά APIs χωρίς ρητή εντολή.
* Μην αλλάζεις credentials.
* Μην αλλάζεις `.env`.
* Αν δουλεύεις σε generator/controller υποβολών, πρώτα εξήγησε τη ροή και μετά πρότεινε patch.

## Τι σημαίνει “ολοκληρώθηκε”

Πριν πεις ότι ολοκληρώθηκε μια εργασία:

1. Δείξε ποια αρχεία άλλαξαν.
2. Δείξε `git diff --stat`.
3. Εξήγησε τι άλλαξε.
4. Πες αν έτρεξες έλεγχο syntax/test.
5. Αν δεν υπάρχουν tests, πες ποιον χειροκίνητο έλεγχο πρέπει να κάνει ο χρήστης.
6. Μην ισχυρίζεσαι ότι είναι production-ready αν δεν έχει ελεγχθεί.
